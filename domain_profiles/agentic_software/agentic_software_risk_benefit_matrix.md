# Agentic Software Risk and Benefit Matrix v0.1

## Benefits

|Benefit|Explanation|
|-|-|
|Preserves accountable human or institutional responsibility|AI agents remain assistive and subordinate to identifiable maintainers, reviewers, release authorities, teams, or institutions.|
|Reduces hidden agentic action|Authorization, scope, evidence references, and expiry are moved upstream before consequential software activity occurs.|
|Improves repository and workflow integrity|Agentic changes to code, tests, configuration, dependencies, CI/CD, and deployment workflows become easier to trace and review.|
|Supports safer AI coding-agent adoption|Teams can use agents where useful without allowing capability to outrun authorization.|
|Clarifies review boundaries|Human review remains meaningful before merge, release, deployment, or production-impacting action.|
|Improves auditability|High-risk agentic software use can leave a proportionate, traceable, time-bounded record.|
|Supports responsible implementation governance|Teams and companies can define which agentic workflows are permitted, restricted, or prohibited.|
|Supports platform-level adoption|Repository platforms and developer tools can map authorization artifacts into workflow controls.|

## Risks

|Risk|Mitigation|
|-|-|
|May be mistaken for software safety, cybersecurity, or compliance certification|State clearly that it is a governance profile, not safety, security, legal, or compliance approval.|
|May create bureaucracy for low-risk work|Apply action categories and keep low-risk assistive uses lightweight.|
|May create false confidence|Clarify that authorization does not prove correctness, security, safety, or production readiness.|
|May be gamed by broad authorization|Require narrow scope, expiry, denied actions, review conditions, and revocation paths.|
|May slow developer workflows|Start with high-consequence actions and automate authorization checks where appropriate.|
|May duplicate existing change-management controls|Frame as a legitimacy layer that supports, not replaces, secure development and change-management processes.|
|May fail across different platforms|Use environment-neutral terms and require local software-environment adaptation.|
|May obscure accountability through multi-agent or tool-chain workflows|Require final accountable human, team, office, or institution for each consequential action class.|
|May expose sensitive information in records|Store references, hashes, metadata, or policy identifiers where appropriate rather than sensitive content.|
|May become vendor-captured|Preserve public inspectability, non-exclusive licensing, attribution, and no exclusive control over the constitutional core.|

## Conclusion

The profile does not eliminate agentic software risk. It makes consequential AI-assisted software action easier to authorize, scope, trace, review, deny, revoke, roll back, and govern within its defined boundaries.
